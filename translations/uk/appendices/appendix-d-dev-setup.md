## 1. Асемблер: sjasmplus

Кожен приклад коду в цій книзі написаний для **sjasmplus** — відкритого макроасемблера Z80/Z80N від z00m128. Він підтримує повний набір інструкцій Z80, включно з усіма індексними режимами IX/IY, макросами, скриптами Lua, множинними форматами виводу та виразами, які роблять написання коду для демосцени практичним.

### Встановлення зі сирців

Найнадійніший спосіб отримати sjasmplus — зібрати його зі сирців. Це гарантує, що у тебе відома версія, та уникає проблем із пакуванням для конкретних платформ.

```bash
git clone https://github.com/z00m128/sjasmplus.git
cd sjasmplus
make
```

На macOS потрібні інструменти командного рядка Xcode (`xcode-select --install`). На Linux потрібні `g++` і `make` (встанови через пакетний менеджер). На Windows використовуй MinGW або WSL.

Після збірки скопіюй бінарний файл `sjasmplus` кудись у PATH:

```bash
# macOS / Linux
sudo cp sjasmplus /usr/local/bin/

# Verify
sjasmplus --version
```

Ти маєш побачити версію 1.20.x або новішу. Цю книгу розроблено й протестовано з v1.21.1.

### Закріплення версії

Репозиторій книги закріплює sjasmplus як git-підмодуль у `tools/sjasmplus/`. Якщо клонувати репозиторій з `--recursive`, ти отримаєш точну версію, використану для компіляції кожного прикладу:

```bash
git clone --recursive https://github.com/[repo]/antique-toy.git
cd antique-toy/tools/sjasmplus
make
```

Це найбезпечніший підхід. Поведінка асемблера може змінюватися між версіями — вираз, що працює у 1.21, може парситися інакше у 1.22.

### Ключові прапорці

| Прапорець | Призначення | Приклад |
|-----------|-------------|---------|
| `--nologo` | Придушити стартовий банер | `sjasmplus --nologo main.a80` |
| `--raw=FILE` | Вивести необроблений бінарник (без заголовка) | `sjasmplus --raw=output.bin main.a80` |
| `--sym=FILE` | Записати файл символів (для зневаджувачів) | `sjasmplus --sym=output.sym main.a80` |
| `--fullpath` | Показувати повні шляхи файлів у повідомленнях про помилки | Корисно з problem matcher у VS Code |
| `--msg=war` | Придушити інформаційні повідомлення, показувати тільки попередження та помилки | Чистіший вивід збірки |
| `--syntax=abf` | Увімкнути всі синтаксичні можливості (A як псевдонім акумулятора, дужки для непрямої адресації, повні псевдоінструкції) | Рекомендовано для початківців; дозволяє `add a, b` поряд з `add b` |

Типова команда збірки для прикладу з розділу:

```bash
sjasmplus --nologo --raw=build/example.bin chapters/ch01-thinking-in-cycles/examples/timing.a80
```

### Розширення файлів

Усі файли вихідного коду Z80 у цій книзі мають розширення `.a80`. Це домовленість, не вимога — sjasmplus не звертає уваги на розширення. Ми використовуємо `.a80`, бо воно розпізнається розширенням Z80 Macro Assembler для VS Code і відрізняє наш код від інших діалектів асемблера.

### Шістнадцяткова нотація

У книзі використовується `$FF` для шістнадцяткових значень. sjasmplus також приймає `#FF` та `0FFh`, але `$FF` — домовленість, прийнята скрізь. Окремий символ `$` позначає поточну адресу лічильника команд, що використовується в конструкціях типу `jr $` (нескінченний цикл) або `dw $ + 4`.

## 2. Редактор: VS Code

Підійде будь-який текстовий редактор. Ми рекомендуємо **Visual Studio Code** через його Z80-специфічні розширення та вбудований термінал. Увесь процес роботи — редагування, збірка, зневадження — відбувається в одному вікні.

### Основні розширення

Встанови їх з маркетплейсу розширень VS Code (Ctrl+Shift+X):

| Розширення | Автор | Що робить |
|------------|-------|-----------|
| **Z80 Macro Assembler** | mborik (`mborik.z80-macroasm`) | Підсвічування синтаксису, автодоповнення, розпізнавання символів для асемблера Z80. Розуміє синтаксис sjasmplus, включно з макросами та локальними мітками. |
| **Z80 Assembly Meter** | Nestor Sancho | Показує кількість байтів і вартість у тактах (T-state) виділених інструкцій у рядку стану. Виділи блок — і одразу бачиш його загальну вартість. Незамінний для підрахунку тактів. |
| **DeZog** | Maziac | Зневаджувач Z80. Підключається до емуляторів або власного вбудованого симулятора. Точки зупину, спостереження за регістрами, інспекція пам'яті. Див. Секцію 4. |

### Задача збірки

Налаштуй задачу збірки, щоб Ctrl+Shift+B компілював поточний файл. Створи `.vscode/tasks.json` у корені проєкту:

```json
{
  "version": "2.0.0",
  "tasks": [
    {
      "label": "Assemble Z80",
      "type": "shell",
      "command": "sjasmplus",
      "args": [
        "--fullpath",
        "--nologo",
        "--msg=war",
        "${file}"
      ],
      "group": {
        "kind": "build",
        "isDefault": true
      },
      "problemMatcher": {
        "owner": "z80",
        "fileLocation": "absolute",
        "pattern": {
          "regexp": "^(.*)\\((\\d+)\\):\\s+(error|warning)[^:]*:\\s+(.*)$",
          "file": 1,
          "line": 2,
          "severity": 3,
          "message": 4
        }
      }
    }
  ]
}
```

`problemMatcher` розбирає вивід помилок sjasmplus, тож клік на помилку в терміналі переходить до відповідного рядка. Прапорець `--fullpath` забезпечує абсолютні шляхи файлів, що потрібно VS Code для правильного розпізнавання.

### Рекомендовані налаштування

Додай ці параметри до `.vscode/settings.json` робочого простору для зручної роботи з Z80:

```json
{
  "editor.tabSize": 8,
  "editor.insertSpaces": false,
  "editor.rulers": [80],
  "files.associations": {
    "*.a80": "z80-macroasm"
  }
}
```

Розмір табуляції 8 із реальними табами відповідає традиційній асемблерній домовленості, де мнемоніки та операнди вирівнюються по фіксованих стовпцях.

## 3. Емулятори

Тобі потрібен емулятор для запуску зібраного коду. Різні емулятори слугують різним цілям.

### ZEsarUX — багатофункціональне зневадження

**ZEsarUX** від cerebellio — найфункціональніший відкритий емулятор ZX Spectrum. Він підтримує повний спектр моделей Spectrum (48K, 128K, +2, +3, Pentagon, Scorpion, ZX-Uno, ZX Spectrum Next), має вбудований зневаджувач та інтегрується з DeZog для зневадження у VS Code.

**Встановлення:**

- macOS: `brew install zesarux`
- Linux: зібрати зі сирців або використати AppImage з https://github.com/chernandezba/zesarux
- Windows: завантажити інсталятор з сайту ZEsarUX

**Чому ZEsarUX для цієї книги:** більшість прикладів з розділів орієнтовані на ZX Spectrum 128K. ZEsarUX емулює перемикання банків пам'яті 128K, звук AY, TurboSound (два чіпи AY) та шаблони конфлікту доступу різних моделей. Його вбудований зневаджувач показує регістри, пам'ять і дизасемблювання без потреби у VS Code. А інтеграція з DeZog забезпечує повноцінне зневадження у VS Code, коли це потрібно.

**Швидкий запуск:**

```bash
# Run a .sna snapshot
zesarux --machine 128k --nosplash output.sna

# Run a .tap file
zesarux --machine 128k --nosplash output.tap
```

### Fuse — легковагий і точний

**Fuse** (Free Unix Spectrum Emulator) від Philip Kendall — легковагий, потактово точний і доступний на кожній платформі. Це найкращий вибір для швидкого тестування, коли тобі не потрібен повний зневаджувач.

**Встановлення:**

- macOS: `brew install fuse-emulator`
- Linux: `apt install fuse-emulator-sdl` (Debian/Ubuntu) або `dnf install fuse-emulator` (Fedora)
- Windows: завантажити з сайту Fuse

**Швидкий запуск:**

```bash
# Run with Pentagon timing (matches the book's T-state counts)
fuse --machine pentagon output.sna

# Run as 128K Spectrum
fuse --machine 128 output.tap
```

Fuse особливо добрий для тестування коду, чутливого до таймінгу, бо його тактова точність добре підтверджена. Якщо твоя тестова обв'язка бордюрних смуг (Розділ 1) показує різні результати у Fuse порівняно з іншим емулятором, довіряй Fuse.

### Unreal Speccy — Windows, орієнтований на Pentagon

Якщо ти розробляєш переважно на Windows і орієнтуєшся на таймінг Pentagon, **Unreal Speccy** — потужний вибір. Він має вбудований зневаджувач із картою пам'яті, точками зупину та моніторингом регістрів AY. Він точно емулює апаратне забезпечення Pentagon і Scorpion.

Завантажити з http://dlcorp.nedopc.com/viewforum.php?f=27 або шукай "Unreal Speccy Portable".

### Для Agon Light 2

Agon Light 2 використовує процесор eZ80 та іншу апаратну архітектуру. Розділ 22 детально розглядає розробку для Agon. Для емуляції **Fab Agon Emulator** надає програмну симуляцію апаратного забезпечення Agon (eZ80 + ESP32 VDP). Він доступний за адресою https://github.com/tomm/fab-agon-emulator і працює на macOS, Linux та Windows.

### Який емулятор мені використовувати?

| Ситуація | Рекомендований емулятор |
|----------|------------------------|
| Повсякденна розробка, запуск прикладів | Fuse (швидкий запуск, точний) |
| Зневадження з точками зупину та спостереженням за регістрами | ZEsarUX + DeZog |
| Розробка музики AY/TurboSound | ZEsarUX (найкраща емуляція AY) |
| Перевірка таймінгу Pentagon | Fuse або Unreal Speccy |
| Розробка для Agon Light 2 | Fab Agon Emulator |
| Швидка перевірка на Windows | Unreal Speccy |

## 4. Зневаджувач: DeZog

**DeZog** від Maziac — це розширення VS Code, яке перетворює твій редактор на зневаджувач Z80. Він підключається до ZEsarUX, CSpect або власного вбудованого симулятора Z80 і забезпечує зневадження, яке очікують сучасні розробники: точки зупину, покрокове виконання, спостереження за регістрами, інспекція пам'яті, перегляд дизасемблювання та стек викликів.

Розділ 23 обговорює DeZog у контексті ШІ-асистованої розробки. Ця секція розглядає практичне налаштування.

### Встановлення

1. Відкрий VS Code.
2. Перейди до Розширень (Ctrl+Shift+X).
3. Знайди "DeZog" від Maziac.
4. Натисни «Встановити».

### Підключення до ZEsarUX

DeZog спілкується з ZEsarUX через сокетне з'єднання. Спершу запусти ZEsarUX із увімкненим ZRCP-сервером (ZEsarUX Remote Control Protocol):

```bash
zesarux --machine 128k --enable-remoteprotocol --remoteprotocol-port 10000
```

Потім створи `.vscode/launch.json` у проєкті:

```json
{
  "version": "0.2.0",
  "configurations": [
    {
      "type": "dezog",
      "request": "launch",
      "name": "DeZog + ZEsarUX",
      "remoteType": "zesarux",
      "zesarux": {
        "port": 10000
      },
      "sjasmplus": [
        {
          "path": "build/output.sld"
        }
      ],
      "topOfStack": "$FF00",
      "commandsAfterLaunch": [
        "-sprites disable",
        "-patterns disable"
      ]
    }
  ]
}
```

Секція `sjasmplus` вказує на файл `.sld` (Source Level Debug), який генерує sjasmplus із прапорцем `--sld=FILE`. Це дає DeZog зневадження на рівні вихідного коду — точки зупину на рядках коду, а не тільки на адресах.

Щоб згенерувати файл `.sld`, додай прапорець до команди збірки:

```bash
sjasmplus --nologo --raw=build/output.bin --sld=build/output.sld --sym=build/output.sym main.a80
```

### Використання внутрішнього симулятора

Для швидкого зневадження без запуску зовнішнього емулятора DeZog включає вбудований симулятор Z80. Зміни `launch.json` на:

```json
{
  "type": "dezog",
  "request": "launch",
  "name": "DeZog Internal Simulator",
  "remoteType": "zsim",
  "zsim": {
    "machine": "spectrum",
    "memoryModel": "ZX128K"
  },
  "sjasmplus": [
    {
      "path": "build/output.sld"
    }
  ],
  "topOfStack": "$FF00"
}
```

Внутрішній симулятор швидше запускається і не вимагає встановленого ZEsarUX. Йому бракує точної емуляції конфлікту доступу, тому не використовуй його для зневадження, критичного до таймінгу — але для логічного зневадження (чи видає моя підпрограма множення правильний результат?) він ідеальний.

### Ключові можливості DeZog

**Точки зупину.** Клікни на поле поруч із рядком коду, щоб встановити точку зупину. Виконання зупиняється, коли лічильник команд Z80 досягає цієї адреси. Ти також можеш встановити умовні точки зупину (наприклад, зупинитися, коли `A == $FF`).

**Спостереження за регістрами.** Панель змінних показує всі регістри Z80: AF, BC, DE, HL, IX, IY, SP, PC та альтернативний набір (AF', BC', DE', HL'). Окремі прапорці (C, Z, S, P/V, H, N) виділені для зручного читання.

**Інспекція пам'яті.** Панель пам'яті показує шістнадцятковий дамп будь-якого діапазону адрес. Ти можеш ввести адресу і побачити, що там знаходиться. Незамінна для перевірки таблиць підстановки, вмісту екранної пам'яті та стану стеку.

**Перегляд дизасемблювання.** Навіть без зневадження на рівні вихідного коду DeZog дизасемблює код навколо поточного PC. Корисно для розуміння того, як насправді виглядає самомодифікований код (SMC) під час виконання.

**Стек викликів.** DeZog відстежує пари CALL/RET і показує стек викликів. Це працює для звичайного коду. Самомодифікований код (SMC) та RET-ланцюжок (Розділ 3) збиватимуть трекер стеку — це очікувана поведінка.

## 5. Збірка прикладів з книги

### Клонування репозиторію

```bash
git clone --recursive https://github.com/[repo]/antique-toy.git
cd antique-toy
```

Прапорець `--recursive` підтягує підмодуль sjasmplus у `tools/sjasmplus/`. Якщо ти вже клонував без нього:

```bash
git submodule update --init --recursive
```

### Передумови

Тобі потрібні `make` та компілятор C++ (для збірки sjasmplus із підмодуля). На більшості систем вони вже є:

- macOS: `xcode-select --install`
- Debian/Ubuntu: `sudo apt install build-essential`
- Fedora: `sudo dnf install make gcc-c++`
- Windows: використовуй WSL або встанови MinGW та GNU Make

### Команди збірки

Makefile проєкту бере все на себе. Весь скомпільований вивід потрапляє у `build/`, яка додана до gitignore.

| Команда | Що робить |
|---------|-----------|
| `make` | Скомпілювати всі приклади розділів за допомогою sjasmplus |
| `make test` | Зібрати всі приклади, звітувати про проходження/провал кожного |
| `make ch01` | Скомпілювати тільки приклади Розділу 1 |
| `make ch04` | Скомпілювати тільки приклади Розділу 4 |
| `make demo` | Зібрати супровідну демо «Antique Toy» |
| `make clean` | Видалити всі артефакти збірки |

Успішний запуск `make test` виглядає так:

```
  OK  chapters/ch01-thinking-in-cycles/examples/timing.a80
  OK  chapters/ch04-maths/examples/multiply.a80
  OK  chapters/ch05-wireframe-3d/examples/cube.a80
  ...
---
12 passed, 0 failed
```

Якщо якийсь приклад не проходить, вивід показує `FAIL` із назвою файлу. Запусти файл із помилкою вручну через sjasmplus, щоб побачити детальну інформацію:

```bash
sjasmplus --nologo chapters/ch04-maths/examples/multiply.a80
```

### Запуск прикладу

Після компіляції завантаж вихідний бінарник в емулятор. Точний метод залежить від формату виводу:

```bash
# If the example produces a .sna snapshot
fuse --machine pentagon build/ch01-thinking-in-cycles/examples/timing.sna

# If you built a raw binary, you need to create a .tap or .sna first,
# or load it at the correct address in the emulator's debugger
```

Більшість прикладів з розділів використовують ORG `$8000` і генерують необроблені бінарники. Щоб запустити їх, або:

1. Використай обгортку `.tap` (Makefile генерує їх, коли вихідний код містить відповідні директиви), або
2. Завантаж бінарник за адресою `$8000` у зневаджувачі емулятора та встанови PC на `$8000`.

## 6. Структура проєкту для власного коду

Коли ти почнеш писати власний код Z80 поза прикладами з книги, ось рекомендована структура каталогів:

```
my-demo/
  src/
    main.a80          -- entry point, includes other files
    effects/
      plasma.a80      -- individual effect routines
      scroller.a80
    data/
      font.a80        -- DB/DW data tables
      sintable.a80
    lib/
      multiply.a80    -- reusable utility routines
      draw_line.a80
  assets/
    title.scr         -- raw screen files
    music.pt3         -- tracker music
  build/              -- compiled output (gitignored)
  Makefile
  .vscode/
    tasks.json        -- build task
    launch.json       -- DeZog configuration
```

### Мінімальний Makefile

```makefile
SJASMPLUS ?= sjasmplus
BUILD_DIR := build
SJASM_FLAGS := --nologo

.PHONY: all clean run

all: $(BUILD_DIR)/demo.bin

$(BUILD_DIR)/demo.bin: src/main.a80 src/effects/*.a80 src/data/*.a80 src/lib/*.a80
	@mkdir -p $(BUILD_DIR)
	$(SJASMPLUS) $(SJASM_FLAGS) --raw=$@ --sym=$(BUILD_DIR)/demo.sym --sld=$(BUILD_DIR)/demo.sld $<

run: $(BUILD_DIR)/demo.bin
	fuse --machine 128 $(BUILD_DIR)/demo.sna

clean:
	rm -rf $(BUILD_DIR)
```

Ключові моменти:

- Головний файл вихідного коду використовує директиви `INCLUDE` для підключення інших файлів. sjasmplus розв'язує include відносно каталогу вихідного файлу.
- Прапорець `--sym` генерує файл символів для довідки. Прапорець `--sld` генерує дані зневадження на рівні вихідного коду для DeZog.
- Перелічи всі включені вихідні файли як залежності, щоб `make` перезбирав при зміні будь-якого файлу.

### Домовленість про Include

У твоєму `main.a80`:

```z80
    ORG $8000

    ; --- Entry point ---
start:
    di
    ; set up stack, interrupts, etc.
    ld   sp, $FF00
    ; ...

    include "lib/multiply.a80"
    include "effects/plasma.a80"
    include "data/sintable.a80"
```

sjasmplus обробляє файли `INCLUDE` вбудовано, ніби текст було вставлено в цю точку. Тримай include впорядковано: спершу бібліотечні підпрограми, потім ефекти, потім дані (оскільки дані зазвичай розміщуються наприкінці бінарника).

## 7. Альтернативні інструменти

Ця книга використовує sjasmplus, бо це найпотужніший асемблер Z80 для роботи над демосценою та розробкою ігор. Але ти можеш зустріти інші інструменти в спільноті.

### Інші асемблери

| Асемблер | Примітки |
|----------|----------|
| **z80asm** | Частина пакету крос-компілятора C z88dk. Добрий, якщо поєднуєш C та асемблер. Інші синтаксичні домовленості. |
| **RASM** | Від Roudoudou. Швидкий, підтримує CPC та Spectrum. Популярний у сцені Amstrad CPC. |
| **pasmo** | Простий, портативний, обмежені можливості. Підходить для невеликих самостійних програм, але не має макросів та розширених можливостей, необхідних для більших проєктів. |

Приклади з книги використовують специфічні можливості sjasmplus (локальні мітки з `.`, від'ємні значення DB, префікс `$` для шістнадцяткових чисел), які можуть не працювати без змін в інших асемблерах. Якщо хочеш портувати приклад на інший асемблер, зміни зазвичай незначні: синтаксис міток, шістнадцяткова нотація та назви директив.

### Інші розширення VS Code

| Розширення | Примітки |
|------------|----------|
| **SjASMPlus Syntax** | Альтернативне підсвічування синтаксису, налаштоване спеціально для sjasmplus. Спробуй, якщо `z80-macroasm` неправильно підсвічує якусь специфічну для sjasmplus можливість. |
| **Z80 Debugger** (Spectron) | Старіший зневаджувач Z80 для VS Code. DeZog його здебільшого замінив. |

### CSpect — емулятор, орієнтований на Next

Якщо у тебе є ZX Spectrum Next або ти орієнтуєшся на специфічні можливості Next (copper, layer 2, DMA, турборежим 28 МГц), **CSpect** від Mike Dailly — еталонний емулятор. DeZog підключається до CSpect так само, як до ZEsarUX. CSpect працює тільки на Windows, але запускається через Wine на macOS та Linux.

### SpectrumAnalyzer

Браузерний інструмент, що візуалізує розкладку екранної пам'яті ZX Spectrum, конфлікти атрибутів та таймінг. Корисний для розуміння обговорення черезрядкової розкладки екрану з Розділу 2. Доступний за адресою https://shiru.untergrund.net/spectrumanalyzer.html (або шукай "ZX Spectrum screen analyzer").

## 8. Вирішення проблем

### "sjasmplus: command not found"

Бінарний файл не знаходиться у PATH. Або скопіюй його до `/usr/local/bin/` (macOS/Linux), або додай його каталог до PATH, або встанови змінну `SJASMPLUS` при запуску make:

```bash
make SJASMPLUS=/path/to/sjasmplus
```

### Помилки компіляції у прикладах з книги

По-перше, переконайся, що використовуєш версію sjasmplus із підмодуля (`tools/sjasmplus/`). Новіші або старіші версії можуть мати іншу поведінку. По-друге, перевір, що збираєш файл із правильного каталогу — sjasmplus розв'язує шляхи `INCLUDE` відносно вихідного файлу, а не робочого каталогу.

### DeZog не може підключитися до ZEsarUX

1. Переконайся, що ZEsarUX запущений з `--enable-remoteprotocol`.
2. Перевір, що номер порту в `launch.json` збігається з аргументом `--remoteprotocol-port`.
3. На macOS може знадобитися дозволити ZEsarUX через файрвол (System Settings > Privacy & Security).
4. Спробуй перезапустити ZEsarUX перед запуском сесії DeZog.

### Емулятор показує спотворений екран

Якщо ти завантажуєш необроблений бінарник і бачиш сміття, найімовірніша причина — неправильна адреса завантаження. Приклади з книги використовують ORG `$8000`. Переконайся, що емулятор завантажує бінарник за цією адресою, а не за `$0000` чи іншою адресою за замовчуванням. Використання виводу `.sna` або `.tap` (що включає інформацію про адресу) уникає цієї проблеми.

### Вивід збірки порожній або нульового розміру

Перевір, що вихідний файл має директиву `ORG` і хоча б одну інструкцію. sjasmplus із `--raw=` створює бінарник від першого випущеного байта до останнього. Якщо нічого не випущено (наприклад, файл містить тільки `ORG $8000` без коду), вивід буде порожнім.
